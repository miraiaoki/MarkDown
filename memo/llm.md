## LLM とは

### 概要

LLM(大規模言語モデル)は ChatGPT などのサービス基盤となる AI 技術です。
実際、LLM が何をしているかというと「これまでの入力を条件として、次にもっとも確率が高いトークンを生成」しています。入力に対して次に来るであろう文章などを**以前の入力情報をもとに**確率的に割り出して出力しているのです。
その為、残念ながら ChatGPT は私たちが入力した内容を完全には理解してくれていません。。

### AI 技術の変遷

#### 1950 年~ ルールベース

文章を処理するために必要な情報を人間が手作業で「この条件ならこう答える」と設定していました。
登録された知識やルールの量と正確さが精度に直結します。

#### 1980~1990 年 統計的手法・初期 NLP

確立や統計を用いて言語を処理する手法です。
特徴量設計は人間が行っていました。

#### 1990~2010 年前半~ 機械学習

機械学習はコンピュータがデータから規則やパターンを学ぶ手法です。
機械学習は正解ラベルをもとに学習する「教師あり学習」、データの構造やパターンを学ぶ「教師なし学習」試行錯誤で最適な行動を学ぶ「強化学習」に分類されます。
データの質と量、特徴のエンジニアリングを工夫することが精度に繋がります。

#### 2012 年 ~深層学習

ニュートラルネットワーク(人間の脳の神経細胞の仕組みを模倣したモデル)を通じてデータの特徴を自動で抽出する手法です。
画像・音声・自然言語処理で飛躍的な性能向上が起きました。

#### 2018 年~LLM

Transformer\* を基盤とした大規模深層学習モデルが誕生しました。
単一モデルで多様な NLP タスクをこなし、入力されたプロンプト（指示・文脈）によって振る舞いが変化します。
※Transformer：「文章中のすべての単語同士の関係を一度に評価できる」Self-Attention を中核にしたニューラルネットワーク構造。単語ごとではなく複数の単語を同時に処理し、文脈的な単語同士の関係を数値的に捉えている。

LLM は
ルール=>統計=>機械学習=>深層学習=>Transformer=>LLM
と成長し、生み出されました。

## LLM についてもう少し詳しく

1. 次トークン予測モデル
   先ほど記載した通り、LLM は入力された文章を理解しているのではなく
   これまでの全トークンの流れから次のトークンを予測し、一番確率の高いものを出力しています。
2. 以前の入力情報を基にする
   LLM は会話の履歴やプロンプト全文を「一続きのトークン列」として扱っています。
   ただ覚えられる量には限界があり、それをコンテキスト長なんて言ったりもします。
   コンテキスト長には入出力データだけではなく、「入力＋過去の会話＋内部生成トークン」が含まれます。
   主要な AI のコンテキスト長は以下の通りです。

   | モデル名         | 提供元     |   最大コンテキスト長 | 特徴・補足                                         |
   | ---------------- | ---------- | -------------------: | -------------------------------------------------- |
   | GPT-4o           | OpenAI     |          128k tokens | 高性能・高速。マルチモーダル対応。実運用で最も安定 |
   | GPT-4.1          | OpenAI     |          128k tokens | 推論・指示追従性が強化された GPT-4 系              |
   | Claude 3 Opus    | Anthropic  |          200k tokens | 長文理解・要約に非常に強い                         |
   | Claude 3 Sonnet  | Anthropic  |          200k tokens | Opus より軽量・高速、実務向け                      |
   | Gemini 1.5 Pro   | Google     | 1M tokens（最大 2M） | 超長文処理が可能（動画・コードベース解析向き）     |
   | Llama 3.1 (70B)  | Meta       |          128k tokens | OSS 系で最大級。自己ホスト可能                     |
   | Llama 3 (8B/70B) | Meta       |     8k / 128k tokens | 軽量〜高性能まで幅広い                             |
   | Mistral Large    | Mistral AI |           32k tokens | 欧州発。高速・コスト効率重視                       |
   | Command R+       | Cohere     |          128k tokens | RAG 最適化、業務文書向け                           |

   コンテキストが長すぎると出力の一貫性と正確性が落ちます。
   主な原因は下記のとおりです。

   - Transformer は全文を一度に見るため、Attention の計算が膨大になってしまう
   - 全体を均等に見ようとして、重要な情報とそうでない情報の重みづけの差が小さくなってしまう
   - 先頭と文末の距離が極端に大きいと、文頭に書いてある制約と現在の入力の関連性が低いと判断され、制約が破られてしまう

3. Transformer 構造と自己注意(Self-Attention)
   Self-Attention とは、各トークンが「過去のどのトークンが重要か」「どの単語に強く依存すべきか」を重み付きで参照します。
   そのため、プロンプト中の書き方・順序・明示性によって Attention の掛が変わります。
4. なぜプロンプトが「思考を誘導」できるのか
   LLM は思考せず予測しています。
   そのため、予測しやすい形式を提示することによって出力結果を誘導することが出来ます。
5. なぜ曖昧なプロンプトはそれっぽい嘘を生むのか　ハルシネーション
   LLM は「分からない」というのが苦手です。
   質問されたら何かを答えるというパターンが多いため、条件があいまいだと「最もそれっぽい文章」を生成するのです。

## 【結論】プロンプトが重要なのはなんで？

LLM の限界を補う手段だからです。
LLM の限界とは、状態を永続的に持てなかったり真偽検証ができない、ゴールを自律的に定義できないことです。
⇒ なので人間が「目的」「前提」「制約」「出力形式」を明示的に与える必要があります。

## ChatGPT を例に

本人に聞いてしまえ！ということで
ChatGPT に「高確率で正しい情報を得る方法のベストプラクティスを提示して」と聞いてみました。
その回答を以下に記載します。（AI の回答を読みやすいように少し加工しています）

---

とても良い締めの問いです。
ここまでの理解（Transformer／Attention／位置エンコーディング／コンテキスト長）を踏まえると、
**「ChatGPT で高確率で正しい情報を得る方法」**は**経験則ではなく工学的設計問題**になります。

以下に **原理 → 具体策 → その理由** の順で、
**再現性の高いベストプラクティス**を提示します。

---

### 結論（設計原則）

> **ChatGPT を「知識検索エンジン」として使うと失敗し、
> 「確率的推論エンジン」として使うと成功する**

その前提で設計します。

---

### 1️⃣ ハルシネーションを抑える最重要原則

#### ❌ やってはいけないこと

- 曖昧な質問
- 前提を省略
- 一度に多くを聞く

#### ⭕ やるべきこと（原理）

> **モデルに「知らないこと」を答えさせない**

#### 実践テンプレ

```text
以下について、事実ベースで説明してください。
不確かな場合は「分からない」と明示してください。
推測や一般論は書かないでください。
```

##### なぜ効くか

- 出力空間を制約
- 「それっぽい文章」の確率を下げる

---

### 2️⃣ 正確性を上げる質問設計（Prompt 設計）

#### 原則：問いは「狭く・具体的に」

##### ❌ 悪い例

```
Transformerを教えて
```

##### ⭕ 良い例

```
Transformerにおける
Self-Attentionの計算対象と
位置エンコーディングの役割を
数式レベルではなく概念レベルで説明してください
```

##### 技術的理由

- 条件付き確率が一点に集中
- 無関係な知識が出にくくなる

---

### 3️⃣ トークン長・コンテキスト長のベストプラクティス

#### 原則

> **長さ ≠ 正確さ
> 情報密度 ≫ 情報量**

#### ⭕ 推奨運用

| 状況             | 対応             |
| ---------------- | ---------------- |
| 会話が長くなった | 新スレッドで再掲 |
| 前提が重要       | 毎回明示         |
| 長文資料         | 要約してから投入 |
| 複数論点         | 分割して質問     |

##### なぜか

- Attention 希釈を防ぐ
- 位置情報の弱体化を防ぐ

---

### 4️⃣ 前提条件・制約は「毎回」書く

#### テンプレ例

```text
前提：
- 対象：ソフトウェアエンジニア
- レベル：基礎〜中級
- 観点：技術的・実装寄り
- 禁止：比喩表現、一般論
```

##### 技術的理由

- トークン位置を最新化
- Attention 重みを復活させる

---

### 5️⃣ 「役割（Role）」を明示する

#### なぜ役割指定が効くのか

- 出力文体・知識領域・抽象度が
  **条件として固定される**

#### 例

```text
あなたは機械学習の研究者です。
以下の質問に学術的観点で答えてください。
```

👉 「何者として答えるか」は
**モデルの探索空間を劇的に狭める**

---

### 6️⃣ 「正しさ」を検証させる

#### 自己検証プロンプト（強力）

```text
上記の回答について、
誤りや不正確な点がないか
技術的観点から自己レビューしてください。
```

##### なぜ効くか

- 1 回目と異なる Attention パターン
- 矛盾が顕在化しやすい

---

### 7️⃣ 不確実領域では「選択肢形式」にする

#### ❌

```
どれが正解？
```

#### ⭕

```
以下の選択肢のうち、
一般的に正しいとされるものを選び、
理由を述べてください。

A. ...
B. ...
C. ...
```

##### 理由

- 出力自由度を強制的に下げる
- ハルシネーション余地を減らす

---

### 8️⃣ 長文・専門資料は「RAG 的」に扱う

#### 原則

> **ChatGPT に“覚えさせない”
> 必要な部分だけ“見せる”**

#### 手順

1. 資料を分割
2. 関係箇所だけ貼る
3. 出典を明示

```text
以下の文章を根拠として回答してください。
この文章に書かれていないことは答えないでください。
```

---

### 9️⃣ 「分からない」を許可する

これは非常に重要です。

#### 明示的に書く

```text
不明な点は無理に補完せず、
「不明」と書いてください。
```

##### なぜか

- LLM は沈黙より生成を選びやすい
- 許可しないと「嘘」が出る

---

### 10️⃣ 実務向け 黄金テンプレ（保存推奨）

```text
あなたは〇〇の専門家です。

前提：
- 対象：
- 観点：
- 制約：

質問：
（具体的・単一論点）

条件：
- 不確実な点は「不明」と明示
- 推測はしない
- 根拠がある場合は明示
```

---

### 最終まとめ（本質）

> **ChatGPT の精度は
> モデル性能ではなく
> 「情報の与え方」で決まる**

- ハルシネーションは**仕様**
- コンテキスト長は**諸刃の剣**
- 正確性は**設計で作る**
